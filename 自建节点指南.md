# 自建节点说明

有了订阅链接生成工具后，其实节点其实已经简化成了`vless+ProxyIP`，决定自己参考各大佬的代码，自己实现一个节点
1. 学习`JavaScript Streams API`
2. 简化定制代码，应当不容易报`1101`错误（一定学习`https://blog.cmliussss.com/p/CF1101/`）
3. 需要泄漏`uuid`和`sni`才能使用。（`vless+reality`其实有泄漏变免费`ProxyIP`风险）
4. `path`需要知道格式才能享受`ProxyIP`否则体验太差
6. 完全不需要对不同协议开发授权认证，因为`TLS`已经对`URL`加密了，可以用来对`wss`授权认证和协议区分，也不限制一定要使用`uuid`

## 如何本地测试？
1. `v2rayN`真连测试地址设置`http://www.msftconnecttest.com/connecttest.txt`
2. 系统代理：`自动配置系统代理`,路由：`全局`
3. `npm run dev`
4. 访问链接`http://127.0.0.1:8787/芝麻开门`，将两条链接导入`v2rayN`
5. `VLESS`节点，将端口(port)由`443`改成`8787`，将传输层安全(TLS)由`tls`改成空
6. `Trojan`节点，必须使用`tls`,本地测试麻烦就不测了
7. `ss+无tls+无加密`节点, `v2rayN`和`Clash Mi`都可以连通成功

## 如何CF节点测试?
1. 本地测试成功后，将代码上传到`Pages`或`Workers`, 记录下地址`xxx.pages.dev`或`xxx.workers.dev`(开启ech后也能使用)
2. 访问链接`https://xxx.workers.dev/芝麻开门`，将两条链接导入`v2rayN`
3. 测试`VLESS`节点和`Trojan`节点是否可用。这两链接的区别是`path=/芝麻开门/vless`或`path=/芝麻开门/trojan`
4. `ss+tls+无加密`节点。`v2rayN`成功(EchConfigList: `cloudflare-ech.com+https://223.5.5.5/dns-query`可使用`workers.dev`域名)。`Clash Mi`失败，原因未知，欢迎网友告知原因

## 迭代升级
用了几个小时后，发现完全一样的`ProxyIP`，但体验非常差。总结了一下
1. `connect`没有设置超时，所以遇到不能连的IP，要花很长时间等待。需要加超时机制
2. 同一个域名下面挂着非常多的IP, `connect`连向了哪个IP是不可控的，非常有可能每次都是无效IP

所以，`DNS解析版`出现了
1. 加了DNS缓存和淘汰机制。所以优秀的节点会一直留在池中，也会被优先选中,失效的才移除，占内存小可忽略
2. 同时还支持部署多个`ProxyIP`。例如：`A={colo}.proxyip.cmliussss.net:443&A=proxyip.hk.cmliussss.net:443`
3. `设置`->`变量和机密`->文本 变量名`UUID`->任意填写，例如：`欢迎使用CF机场`

## 对`ProxyIP`的获取也支持多种方式
1. `ip=221.156.228.221:443`或`ip=[2602:fc59:11:64::6810:1]:443`。此设置不解析、不参与淘汰。
2. `A=proxyip.cmliussss.net:443`,使用DNS中`A`记录，即`IPv4`地址
3. `AAAA=proxyip.cmliussss.net:443`,使用DNS中`AAAA`记录，即`IPv6`地址
4. `TXT=kr.william.us.ci`,使用DNS中`TXT`记录。使用此功能一般是为了指定端口号
5. `url=https://raw.githubusercontent.com/hofccyf/myip/refs/heads/main/us.txt`,使用网页中的IP和端口

- `IPv6`必有`[]`用于区分端口号。
- 确信某域名下只有少量IP，并且都非常稳定，可以这样写`ip=proxyip.example.com:443`,它将不会主动`DNS`解析，不淘汰，而是直接使用

本人常用配置为`/芝麻开门/trojan?A={colo}.proxyip.cmliussss.net:443`

## 如何固定出口IP
需要配合`ip`和`skip`两个参数。例如：`ip=proxyip.example.com:443&skip=true`

原始目标集合由两部分组成：`[目标网站,ProxyIP]`，加了`skip=true`后，将变成`[ProxyIP]`，实现固定`出口IP`的功能。类似于`DNS`中为`VPS`开启小黄云

## NAT64服务也可以做ProxyIP
可以用公开的`NAT64`服务做指向`CF的IPv4`地址。公益服务极少，千万不要滥用，只用于保底
- `https://www.cloudflare.com/zh-cn/ips`可以查到`IPv4`，例如：`104.16.0.0`可转为`6810:0000`
- `https://nat64.xyz/`可查到`nat64`公益服务。例如：`2602:fc59:11:64::/96`

可以依此得到`nat64`的`IPv6`地址。也可以将IPv4地址和IPv6前缀提供给AI，可直接返回新地址
```javascript
'2602:fc59:11:64::'+'104.16.0.1'.split('.').map(p => Number(p).toString(16).padStart(2, '0')).join('').match(/.{4}/g).join(':')
// CF的范播地址
const cf_anycast = ['108.162.192.0/18', '190.93.240.0/20', '188.114.96.0/20', '104.16.0.0/13', '104.24.0.0/14', '172.64.0.0/13']
```
这样参数可以填写为`ip=[2602:fc59:11:64::6810:1]`,也可以将这些少量的`nat64`地址提前登记到某个域名下的`AAAA`中，例如：`AAAA=nat64.xxx.xyz`

对于`nat64`这类端口完全对应的情况，不用填写端口号，将自动填充目标网站端口号，而只开放特定端口的代理一定要写端口号

## 搭建属于自己的cloudflare优选IP和ProxyIP
容易混淆的地方，根据甬哥的视频教程，区别如下
1. 优选IP：`v2rayN`中`address`填写的地方。它可以是`CF的IP`,也可以是`非CF的IP`
2. ProxyIP：`v2rayN`中`path`中`/?ip=xxx`填写的地方。必须是`非CF的IP`。不要浪费有限的`ProxyIP`资源把`ProxyIP`当做`优选IP`使用

它们使用`http://ip:port/cdn-cgi/trace`访问，显示400错误有`cloudflare`，使用`https://domain:port/cdn-cgi/trace`访问，显示`trace信息`

ProxyIP有两种搭建方案
1. iptables系列：https://blog.tanglu.me/cloudflare_proxy_ip
2. vless+reality: https://ygkkk.blogspot.com/2023/07/cfworkers-vless.html#教程八

推荐使用`https://github.com/yonggekkk/x-ui-yg`搭建`vless+reality`。只要将`reality`的域名设置成`time.is`、`ip.sb`等域名，就是`ProxyIP`，否则只是普通的`vless`节点。

域名是未被屏蔽的CF托管的官方域名，例如优选域名中各大厂的域名。

注意：PrxoyIP都有被扫，然后被分享的可能。所以，不要使用常见端口，推荐使用IPv6，时常关注VPS流量消耗。

## 禁用智能多宿主名称解析
```bash
reg add "HKLM\SOFTWARE\Policies\Microsoft\Windows NT\DNSClient" /v DisableSmartNameResolution /t REG_DWORD /d 1 /f
ipconfig /flushdns
```
