# 自建节点说明

有了订阅链接生成工具后，其实节点其实已经简化成了`vless+ProxyIP`，决定自己参考各大佬的代码，自己实现一个节点
1. 学习`JavaScript Streams API`
2. 简化定制代码，应当不容易报`1101`错误（一定学习`https://blog.cmliussss.com/p/CF1101/`）
3. 需要泄漏`uuid`和`sni`才能使用。（`vless+reality`其实有泄漏变免费`ProxyIP`风险）
4. `path=/`需要知道格式才能享受`ProxyIP`否则体验太差
6. 所以完全不需要做那么多安全检查，效率能大大提升

## 如何本地测试？
1. `v2rayN`真连测试地址设置`http://www.msftconnecttest.com/connecttest.txt`
2. 系统代理：`自动配置系统代理`,路由：`全局`
3. 导入链接：`vless://12345678-1234-1234-1234-123456789012@127.0.0.1:8787?encryption=none&security=none&sni=127.0.0.1&fp=random&type=ws&host=localhost&path=%2F#LocalTest`
4. `npm run dev`

，先让AI实现了一版，本地真连测试`-1`。一步步检查，发现几个关键点：
1. `request.headers.get('sec-websocket-protocol')`，首包居然是通过`headers`获取的
2. `ws.send(new Uint8Array([0, 0])`，需要回复

测试成功后，又遇到非常多的异常，加了`try catch`后代码就膨胀了。还发现在`v2rayN`中内核由`Xray`换成`singbox`后，真连测试能过，但无法上网，未知原因。

## 更短还支持`ProxyIP`的源代码
偶然又在网上找到[shulng/cf-worker-vless](https://github.com/shulng/cf-worker-vless)，简单代码更适合看懂`ProxyIP`原理

## `ProxyIP`
cmliussss大佬提供了两套`ProxyIP`域名，例如：
1. 按数据中心：`hkg.proxyip.cmliussss.net`、`lax.proxyip.cmliussss.net`等
2. 按地区：`proxyip.hk.cmliussss.net`、`proxyip.us.cmliussss.net`、`proxyip.jp.cmliussss.net`等

大佬给的教程是用的按地区，但兜底代码是用的按数据中心。

所以我改了`shulng`的版本,使用`/?ip=colo.proxyip.cmliussss.net:443`,将自动替换`colo`为当前访问的[数据中心](https://www.cloudflarestatus.com/)

可以使用`nslookup -type=A hkg.proxyip.cmliussss.net`、`nslookup -type=TXT kr.william.us.ci`看到实际IP

## 迭代升级
用了几个小时后，发现完全一样的`ProxyIP`，但体验非常差。总结了一下
1. `connect`没有设置超时，所以遇到不能连的IP，要花很长时间等待。需要加超时机制
2. 同一个域名，下面挂着非常多的IP, `connect`连向了哪个IP是不可控的，非常有可能每次都是无效IP

所以，`DNS解析版`出现了
1. 加了DNS缓存和淘汰机制。目前是`ProxyIP池小于30`并`刷新时间大于10分钟`才刷新。所以优秀的节点会一直留在池中，也会被优先选中
2. 同时还支持部署多个`ProxyIP`。例如：`/?A=colo.proxyip.cmliussss.net:443&AAAA=proxyip.hk.cmliussss.net:443`
3. 不同的`url.search`之间`ProxyIP池`是隔离的，失效的才移除，占内存小可忽略
4. `设置`->`变量和机密`->文本 变量名`UUID`->各版的`uuid`(标准的 8-4-4-4-12 十六进制格式)。可通过在线网页生成

## 对`ProxyIP`的获取也支持多种方式
1. `/?ip=221.156.228.221:443`或`/?ip=[2602:fc59:11:64::6810:1]:443`。此设置不解析、不参与淘汰。
2. `/?A=proxyip.cmliussss.net:443`,使用DNS中`A`记录，即`IPv4`地址
3. `/?AAAA=proxyip.cmliussss.net:443`,使用DNS中`AAAA`记录，即`IPv6`地址
4. `/?TXT=kr.william.us.ci`,使用DNS中`TXT`记录。使用此功能一般是为了指定端口号
5. `/?url=https://raw.githubusercontent.com/hofccyf/myip/refs/heads/main/us.txt`,使用网页中的IP和端口

- `IPv6`必有`[]`用于区分端口号
- 确信某域名下只有少量IP，并且都非常稳定，可以这样写`/?ip=proxyip.example.com:443`,它将不会进行`DNS`解析，而是直接使用

本人常用配置为`/?A=colo.proxyip.cmliussss.net:443&AAAA=proxyip.cmliussss.net:443`

## 如何固定出口IP
需要配合`ip`和`skip`两个参数。例如：`/?ip=proxyip.example.com:443&skip=true`

原始目标集合由两部分组成：`[目标网站,ProxyIP,...]`，加了`skip=true`后，将变成`[ProxyIP,...]`，实现固定`出口IP`的功能

## NAT64服务也可以做ProxyIP
可以用公开的`NAT64`服务做指向`CF的IPv4`地址。公益服务极少，千万不要滥用，只用于保底
- `https://www.cloudflare.com/zh-cn/ips`可以查到`IPv4`，例如：`104.16.0.0`可转为`6810:0000`
- `https://nat64.xyz/`可查到`nat64`公益服务。例如：`2602:fc59:11:64::/96`

得到`nat64`的`IPv6`地址
```javascript
'2602:fc59:11:64::'+'104.16.0.0'.split('.').map(p => Number(p).toString(16).padStart(2, '0')).join('').match(/.{4}/g).join(':')
// CF的范播地址
const cf_anycast = ['108.162.192.0/18', '190.93.240.0/20', '188.114.96.0/20', '104.16.0.0/13', '104.24.0.0/14', '172.64.0.0/13']
```
这样参数可以填写为`/?ip=[2602:fc59:11:64::6810:1]`,也可以将这些少量的`nat64`地址提前登记到某个域名下的`AAAA`中，例如：`/?AAAA=nat64.xxx.xyz`

对于`nat64`这类端口完全对应的情况，不用填写端口号，将自动填充目标网站端口号，而只开放特定端口的代理一定要写端口号

## 搭建属于自己的cloudflare优选反代IP和ProxyIP
容易混淆的地方，根据甬哥的视频教程，区别如下
1. 优选反代IP：`v2rayN`中`address`填写的地方。它可以是`CF的IP`,也可以是`非CF的IP`
2. ProxyIP：`v2rayN`中`path`中`/?ip=xxx`填写的地方。必须是`非CF的IP`

它们使用`http://ip:port/cdn-cgi/trace`访问，显示400错误有`cloudflare`，使用`https://domain:port/cdn-cgi/trace`访问，显示`trace信息`

ProxyIP有两种搭建方案
1. iptables系列：https://blog.tanglu.me/cloudflare_proxy_ip
2. vless+reality: https://ygkkk.blogspot.com/2023/07/cfworkers-vless.html。教程八

推荐使用`https://github.com/yonggekkk/x-ui-yg`。只要将`reality`的域名设置成`time.is`、`ip.sb`等域名，就是`ProxyIP`，否则只是普通的`vless`节点。

域名是未被屏蔽的CF托管的官方域名，例如优选域名中各大厂的域名。

注意：PrxoyIP都有被扫，然后被分享的可能。所以，不要使用常见端口，推荐使用IPv6，时常关注VPS流量消耗。
